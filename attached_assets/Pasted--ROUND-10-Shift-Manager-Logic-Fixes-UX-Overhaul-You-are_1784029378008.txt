## TASK 1 вЂ” Support overnight shifts (logic fix, currently a blocker)

In `client/src/pages/ShiftsManager.tsx`, the `shiftFormSchema` currently refines `endTime > startTime` and rejects shifts like 18:00вЂ“02:00.

1. **Remove** the `endTime > startTime` refinement. Keep the "required" validation on both fields.
2. Semantics: if `endTime <= startTime`, the shift ends the **next day**. No schema change вЂ” `startTime`/`endTime` stay as-is in the DB.
3. In the shift dialog, under the time inputs, add muted helper text: `"An end time earlier than the start time means the shift ends the next day."`
4. Everywhere a shift's time range is rendered (`ShiftSchedule.tsx` desktop + mobile cards, and the copy dialog if applicable), append a small `(+1)` suffix when `endTime <= startTime`, e.g. `18:00 - 02:00 (+1)`.

## TASK 2 вЂ” Timezone-safe date arithmetic on the server (latent bug)

In `server/routes.ts`, inside `POST /api/shifts/copy-to-next-week`, the +7-day math currently does `new Date(shift.shiftDate + "T00:00:00")` then `toISOString().substring(0, 10)`. This only works because the server runs UTC.

Replace with pure UTC string arithmetic вЂ” exactly this pattern:

```ts
const [y, m, d] = shift.shiftDate.split("-").map(Number);
const shifted = new Date(Date.UTC(y, m - 1, d + 7));
const newDateStr = shifted.toISOString().substring(0, 10);
```

No other behavior change to this route. The `nextWeekHadShifts` logic and response shape stay identical.
