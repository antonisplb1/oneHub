## FIX 1 — Add a "promotion pass" to the billing reconciler

**File:** `server/billing.ts` (function `reconcileBilling`), plus tests in `server/billing.test.ts`.

**The bug:** `reconcileBilling` only selects candidates with `subscriptionStatus = 'active' AND stripeSubscriptionId IS NOT NULL`, so it can only demote. If the `checkout.session.completed` webhook is missed (network failure, deploy during delivery, etc.), a merchant who PAID stays `inactive` with no `stripeSubscriptionId` forever. The reconciler must also heal in the other direction: Stripe says the customer has a live subscription, but our DB doesn't reflect it.

**Requirements:**

1. After the existing demotion/drift loop, add a second candidate query: users where `stripeCustomerId IS NOT NULL` AND (`subscriptionStatus != 'active'` OR `stripeSubscriptionId IS NULL`). Exclude users with `chargeFree = true` (charge-free accounts must never be promoted into billing).
2. For each candidate, call `stripe.subscriptions.list({ customer: user.stripeCustomerId, status: 'all', limit: 10 })` and look for a subscription whose `status` is in the existing `LIVE_STRIPE_STATUSES` constant (`['active', 'trialing', 'past_due']` — reuse the constant, do not redefine it).
3. If a live subscription is found: set `subscriptionStatus = 'active'` and `stripeSubscriptionId = <that subscription's id>` on the user row, and log a `console.warn` line in the same style as the existing `[Reconcile]` logs (e.g. `[Reconcile] Promoted <email>: live Stripe subscription "<id>" found but account was inactive (webhook likely missed)`).
4. If more than one live subscription is found for the same customer, do NOT auto-fix. Log a `console.error` and record it as an error entry — a human must resolve duplicate subscriptions.
5. **Transient-error guard (critical):** if the `subscriptions.list` call throws for any reason, log the error and skip that user WITHOUT changing anything. A Stripe outage must never cause mass changes in either direction. This mirrors the existing guard on the demotion side.
6. **dryRun:** when `opts.dryRun` is true, report what WOULD be promoted (same pattern as the existing `deactivateDeadAccount` dry-run handling) but change nothing in the DB.
7. Extend the `ReconciliationResult` interface with a `promoted` array (entries shaped like the existing deactivation entries: `userId`, `email`, the found `subscriptionId`, `fixed: boolean`, optional `error`). Include promotion counts in the final `[Reconcile] Finished:` summary log line, matching its existing format.
8. The admin endpoint `POST /api/admin/billing/reconcile` in `server/routes.ts` already returns the `ReconciliationResult` — verify the new `promoted` field flows through without extra work; do not restructure that endpoint.
9. The `BillingStripe` type in `server/billing.ts` will need `subscriptions.list` added to its shape. Update it minimally and update the Stripe spy/mock objects in `server/billing.test.ts` accordingly so existing tests still compile and pass.
10. Do NOT change the existing demotion logic, drift-correction logic, `isBillableAccount`, `expectedPriceForUser`, `syncBillingFromStores`, or any pricing constants.

**New tests to add in `server/billing.test.ts` (follow the existing test style with Stripe spies):**
- Inactive user with `stripeCustomerId` and one live Stripe subscription → promoted: status set to `active`, `stripeSubscriptionId` stored, appears in `result.promoted` with `fixed: true`.
- Inactive user with `stripeCustomerId` but no live subscriptions (empty list, or only `canceled`/`incomplete_expired` ones) → untouched, not in `promoted`.
- Inactive user with `chargeFree = true` → never queried against Stripe, never promoted.
- `dryRun: true` → promotion candidates reported in `result.promoted` with `fixed: false`, DB unchanged (assert the update was never called).
- `subscriptions.list` throws → user untouched, error logged, no DB change.
- Two live subscriptions on the same customer → no DB change, recorded as an error entry.

**Files:** `shared/schema.ts`, a new Drizzle migration in `migrations/`, `server/demoSeed.ts`, `client/src/pages/MenuBuilder.tsx`, `client/src/pages/PublicMenu.tsx`, and any other place that reads/writes `menuItems.price` (search the whole repo for usages before finishing).

**Requirements:**

The migration must convert existing data, not just change the column type: `ALTER TABLE menu_items ALTER COLUMN price TYPE integer USING ROUND(price * 100)::integer;` (adapt to the exact SQL drizzle-kit produces — the critical part is the `USING ROUND(price * 100)` so existing euro floats become cents, never truncated).
3. **API contract:** the API now sends and receives cents. Do not add server-side conversion layers.
4. **Client — `MenuBuilder.tsx`:** the price form field stays a euro decimal input for the merchant (e.g. "9.99"). 7. Search the entire repo (`client/`, `server/`, `shared/`) for any other read/write of menu item `price` (analytics, exports, sorting, etc.) and convert those too. Do not miss the item create AND item update paths.
8. Do NOT touch `rewards.winChance` (it is a probability, `real` is correct) and do NOT touch any subscription/billing pricing.

## FIX 3 — Persist sessions in Postgres instead of memory + remove hardcoded session-secret fallback

**File:** `server/auth.ts` (function `setupAuth`, lines ~60-80).

**The bug (two parts):**

Part A: sessions are stored in `memorystore` (`store: new MemoryStore({ checkPeriod: 86400000 })`). Every server restart or redeploy wipes ALL sessions — every merchant, subuser, and admin is logged out. On Replit hosting, restarts are frequent, so real customers will experience random logouts constantly. The infrastructure for the correct fix already exists and is unused: `connect-pg-simple` is in `package.json` dependencies (v10) and a `sessions` table is already defined in `shared/schema.ts` (line ~157).

Part B: the session secret has a hardcoded fallback: `secret: process.env.SESSION_SECRET || "merchant-hub-secret-key-change-in-production"`. If `SESSION_SECRET` is ever missing in production, session cookies are signed with a publicly known string, meaning anyone can forge a valid session cookie for any user. A missing secret must be a startup failure, never a silent fallback.

**Requirements:**

1. Replace the `memorystore` store with `connect-pg-simple`, using the SAME Postgres connection the app already uses (the Neon serverless pool exported from `server/db.ts`). Do not create a second database connection. Example shape:
   ```ts
   import connectPgSimple from "connect-pg-simple";
   const PgStore = connectPgSimple(session);
   // ...
   store: new PgStore({
     pool,               // the existing pool from ./db
     tableName: "sessions",
     createTableIfMissing: false, // the table already exists via Drizzle schema/migrations
   }),
   ```
2. Verify the column names/types of the existing `sessions` table in `shared/schema.ts` match what `connect-pg-simple` expects (`sid` varchar primary key, `sess` json, `expire` timestamp with an index). If the existing Drizzle table definition differs, adjust the Drizzle table definition + a migration to match connect-pg-simple's expected shape — do NOT let connect-pg-simple create its own parallel table.
3. Remove the `memorystore` import and drop the `memorystore` package from `package.json` dependencies.
4. Fix the secret handling: in production (`app.get("env") === "production"` or `NODE_ENV === "production"`), if `process.env.SESSION_SECRET` is not set, throw an error at startup with a clear message (fail fast). In development only, a fallback is acceptable. Remove the current always-on hardcoded fallback string.
5. Session behavior must be otherwise identical: same cookie settings (`maxAge` 7 days, `httpOnly`, `sameSite: "lax"`, `secure` in production), same `resave: false` / `saveUninitialized: false`, and both the merchant/subuser Passport sessions AND the admin sessions (`req.session.adminId`) must keep working — they share the same session middleware, so one store change covers both, but verify the admin login flow still works after the change.

**Verification:** log in as the demo merchant AND as an admin, restart the dev server (`npm run dev` stop/start), and confirm both sessions survive the restart (no re-login required). Confirm rows appear in the `sessions` table. Confirm the app refuses to start in production mode without `SESSION_SECRET`.
