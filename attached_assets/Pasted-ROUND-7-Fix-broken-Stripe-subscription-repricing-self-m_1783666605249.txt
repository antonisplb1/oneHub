ROUND 7 — Fix broken Stripe subscription repricing (self-managed Product, no manual Stripe setup)

CONTEXT / THE BUG
`updateStripeSubscriptionPrice` in server/billing.ts (line ~84) calls
stripe.subscriptions.update() with `price_data` containing `product_data: { name,
description }`. That inline product_data shape is only valid for Checkout Sessions.
The subscriptions update API rejects it with an "unknown parameter" error — this is
why the code has an `as any` cast. Every reprice call against real Stripe fails:
- applyCustomPrice: error is swallowed (best-effort by design), so admin custom
  prices never reach Stripe.
- reconcileBilling: uses the same broken function, so the drift safety net cannot
  correct anything either.
- syncBillingFromStores: same failure when primary-store products change.
The tests never caught it because the spies in server/billing.test.ts accept the
invalid shape instead of rejecting it like real Stripe does.

REQUIRED CHANGES

1. In server/billing.ts, add and export:

   getOrCreateUnihubProduct(stripe: BillingStripe): Promise<string>

   Behavior:
   a. If a module-level cached product ID exists, return it immediately.
   b. Otherwise call stripe.products.search({ query:
      "metadata['unihub']:'subscription' AND active:'true'", limit: 1 }).
      If a product is found, cache and return its id.
   c. If none found, call stripe.products.create({ name: "uniHub Subscription",
      metadata: { unihub: "subscription" } }), cache and return its id.
   Also export a test-only helper `_resetProductCacheForTests()` that clears the
   module-level cache.

2. Extend the exported BillingStripe interface (line ~75) with:
   products: {
     search(params: any): Promise<any>;
     create(params: any): Promise<any>;
   };

3. Rewrite updateStripeSubscriptionPrice:
   - Call getOrCreateUnihubProduct(stripe) to obtain the product ID.
   - Build price_data as: { currency: 'eur', product: <productId>,
     unit_amount: newPriceCents, recurring: { interval: 'month' } }.
   - REMOVE product_data entirely and REMOVE the `as any` cast — the object must
     type-check against Stripe's types without casting.
   - Pass the per-merchant `description` string as a top-level `description`
     parameter on the same stripe.subscriptions.update() call (subscription-level
     description), since it no longer lives on the product.
   - Keep the function signature, the prorationBehavior parameter, and its
     'create_prorations' default EXACTLY as they are. Keep the retrieve-item-id
     logic and the "Subscription has no items" error unchanged.

   - makeStripeSpy currently records `description` from

   - Call _resetProductCacheForTests() in test setup/between tests so the cache
