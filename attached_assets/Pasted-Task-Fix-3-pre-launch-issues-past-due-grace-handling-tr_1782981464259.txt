FIX 1 — Stop locking out past_due merchants; align access gating with the reconciler

Files: server/routes.ts (requireSubscription middleware), server/billing.ts, server/billing.test.ts, plus a small client banner (see req. 6).

The bug: When a merchant's card payment fails once, Stripe sets the subscription to past_due and fires customer.subscription.updated. The webhook writes that status verbatim into users.subscriptionStatus. The requireSubscription middleware checks subscriptionStatus === "active", so the merchant is locked out of their entire dashboard immediately — even though Stripe's dunning is still automatically retrying the card and usually recovers it. Meanwhile LIVE_STRIPE_STATUSES in server/billing.ts (['active', 'trialing', 'past_due']) already treats past_due as live, so the app's gate and the reconciler currently disagree about what "live" means.

The intended model: past_due = grace state. Access stays ON, the merchant sees a payment-failed warning, and lockout happens only when Stripe gives up and transitions the subscription to canceled or unpaid (those statuses arrive via the existing webhooks — no new logic needed for them).

Requirements:


In server/billing.ts, add an exported pure helper:


ts   // Whether this account's subscription status grants access to the app.
   // past_due is a grace state: Stripe dunning is still retrying the card, so
   // access stays on. Lockout happens only when Stripe gives up (canceled/unpaid).
   export function hasAccessGrantingSubscription(status: string | null | undefined): boolean

It must return true exactly when the status is in LIVE_STRIPE_STATUSES. Reuse the existing constant — do not define a second list. This one function becomes the single source of truth for "does this subscription status grant access."
2. In server/routes.ts, change requireSubscription to use hasAccessGrantingSubscription(req.user!.subscriptionStatus) instead of the === "active" comparison. Keep the chargeFree and trial checks exactly as they are.
3. Audit EVERY other subscriptionStatus === 'active' (and === "active") comparison in server/routes.ts, server/billing.ts, and client/src/ and classify each one:

Billing gating (isBillableAccount, syncBillingFromStores's hasActiveSub check, reconciler candidate queries) → these decide whether to touch Stripe pricing. A past_due subscription is still a real Stripe subscription that should be repriced when products change, and should still be drift-checked by the reconciler. Update these to accept any LIVE_STRIPE_STATUSES status (paired with stripeSubscriptionId present) rather than only 'active'. The reconciler's candidate query (eq(users.subscriptionStatus, 'active')) must become an inArray(users.subscriptionStatus, LIVE_STRIPE_STATUSES) so past_due accounts aren't invisible to drift correction.
